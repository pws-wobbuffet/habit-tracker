## Data model

Defined in `src/types/index.ts`.

```ts
type Schedule =
  | { type: 'daily' }
  | { type: 'weekdays' }
  | { type: 'days'; days: number[] } // 0=Mon, 1=Tue, ..., 6=Sun (Monday-based)
  | { type: 'weekly'; perWeek?: number }

interface HabitTarget {
  unit: string // e.g. 'min', 'L', 'steps'
  qty: number
}

interface Habit {
  id: string
  name: string
  icon: string // emoji
  hex: string // habit color e.g. '#4caf63'
  schedule: Schedule
  isFavorite: boolean
  createdAt: string // "YYYY-MM-DD"
  target?: HabitTarget // optional daily goal
}

interface Completion {
  id: string
  habitId: string
  date: string // "YYYY-MM-DD"
  note?: string
  voiceMemoUrl?: string
}

interface UserProfile {
  name: string
  joinDate: string // "YYYY-MM-DD"
  themeMode: 'light' | 'dark'
  accent: string // hex accent color e.g. '#3b5bdb'
}
```

All dates are stored as `"YYYY-MM-DD"` strings. This makes sorting and comparison trivial (string comparison works correctly for ISO dates) and avoids timezone issues with `Date` objects.

## Storage layer

The storage layer is fully decoupled from the UI via an interface (`src/storage/types.ts`):

```ts
interface StorageBackend {
  getHabits(): Promise<Habit[]>
  putHabit(habit: Habit): Promise<void>
  deleteHabit(id: string): Promise<void>
  getCompletions(filter?: CompletionFilter): Promise<Completion[]>
  putCompletion(completion: Completion): Promise<void>
  deleteCompletion(id: string): Promise<void>
  getProfile(): Promise<UserProfile | null>
  putProfile(profile: UserProfile): Promise<void>
}

interface CompletionFilter {
  habitId?: string
  date?: string
}
```

`src/storage/index.ts` exports a module-level `backend` singleton. It reads `VITE_STORAGE_BACKEND` at startup and returns the matching implementation:

| `VITE_STORAGE_BACKEND` | Class           | File                |
| ---------------------- | --------------- | ------------------- |
| `idb` (default)        | `IDBBackend`    | `storage/idb.ts`    |
| `remote`               | `RemoteBackend` | `storage/remote.ts` |

### IndexedDB backend (`storage/idb.ts`)

Uses the `idb` library. Three object stores:

| Store         | Key  | Indexes                                |
| ------------- | ---- | -------------------------------------- |
| `habits`      | `id` | --                                     |
| `completions` | `id` | `by-habit`, `by-date`, `by-habit-date` |
| `profile`     | `id` | --                                     |

The database opens lazily (first access) and caches the handle. IDB version is `1`; no migration logic is needed for new optional fields since IDB stores full JS objects.

### Remote backend (`storage/remote.ts`)

An HTTP client that talks to the optional Bun server in `server/`. All calls map directly to REST routes:

```
GET    /habits          -> getHabits()
PUT    /habits/:id      -> putHabit()
DELETE /habits/:id      -> deleteHabit()
GET    /completions     -> getCompletions()  (with ?habitId= and ?date= params)
PUT    /completions/:id -> putCompletion()
DELETE /completions/:id -> deleteCompletion()
GET    /profile         -> getProfile()
PUT    /profile         -> putProfile()
```

Requires `VITE_API_URL` to be set at build time.

### Adding a custom backend

Implement `StorageBackend` from `src/storage/types.ts`, then register it in the factory in `src/storage/index.ts`:

```ts
case 'my-backend':
  return new MyBackend()
```

No other changes are needed -- all stores and hooks use the `backend` singleton.

## State management

Four Zustand stores in `src/store/`:

| File             | What it holds                            | Persisted         |
| ---------------- | ---------------------------------------- | ----------------- |
| `habits.ts`      | All habits, hydration state              | Yes (via backend) |
| `completions.ts` | All completions, hydration state         | Yes (via backend) |
| `profile.ts`     | User profile                             | Yes (via backend) |
| `ui.ts`          | Active toast list, active sheet habit ID | No (ephemeral)    |

### Hydration pattern

Every persistent store exposes a `hydrate()` action. `main.tsx` calls all three in parallel on startup:

```ts
await Promise.all([
  useHabitsStore.getState().hydrate(),
  useCompletionsStore.getState().hydrate(),
  useProfileStore.getState().hydrate(),
])
// then render <App />
```

Mutations write through to the backend and update local state optimistically. There is no Zustand `persist` middleware -- individual records are written, not the entire store.

### Selecting from stores

Subscribe to the minimal slice to avoid unnecessary re-renders:

```ts
// Good: re-renders only when `habits` changes
const habits = useHabitsStore((s) => s.habits)

// Also fine: action references are stable
const addHabit = useHabitsStore((s) => s.addHabit)
```

## App shell and routing

`src/App.tsx` is the root component. It renders inside `src/main.tsx` after hydration.

The layout is a responsive split keyed off a 768 px breakpoint (`useIsDesktop()` in `App.tsx`):

```
App  (flex row, full height)
  SideNav            -- desktop only (>= 768px)
  [routed content]
    Route /                -> OverviewScreen (desktop) / TodayScreen (mobile)
                              -> OnboardingScreen when no habits exist yet
    Route /overview        -> OverviewScreen
    Route /create          -> HabitCreatorScreen
    Route /calendar        -> CalendarScreen
    Route /analytics       -> AnalyticsScreen
    Route /achievements    -> AchievementsScreen
    Route /habits          -> HabitsScreen
    Route /settings        -> SettingsScreen
    Route /habit/:id       -> HabitDetailScreen
  FloatingNav        -- mobile only (< 768px), hidden during onboarding
  HabitSheet  (portal on document.body, rendered when activeHabitId != null)
  ToastContainer  (portal on document.body, fixed position)
```

The same `/` route renders `OverviewScreen` on desktop and `TodayScreen` on mobile; the active mode is recomputed live via a `matchMedia('(min-width: 768px)')` listener.

### Navigation

There is no device-frame shell; the app fills the viewport on every screen size. Which navigation renders is chosen at runtime:

- **Desktop (>= 768px):** the left [`SideNav`](/components/#sidenav).
- **Mobile (< 768px):** the floating glass-pill [`FloatingNav`](/components/#floatingnav) at the bottom, hidden during onboarding.

Animation is loaded lazily via Framer Motion's `LazyMotion` (`domAnimation`); individual components animate themselves (e.g. `Sheet`, `Toast`) rather than a global route-transition wrapper.

## PWA

Configuration lives in `vite.config.ts` under `VitePWA(...)`.

- **Strategy**: `generateSW` -- Workbox generates the service worker at build time
- **Update**: `registerType: 'autoUpdate'` -- new SW activates immediately
- **Precache**: all JS, CSS, HTML, icons, and woff2 fonts
- **Runtime cache**: Google Fonts stylesheets (`StaleWhileRevalidate`) and font files (`CacheFirst`, 1-year TTL)
- **Minification**: disabled (`minify: false`) to avoid terser compatibility issues on non-Node runtimes

`index.html` includes the full set of Apple-specific meta tags:

```html
<meta name="apple-mobile-web-app-capable" content="yes" />
<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent" />
<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover" />
```

`viewport-fit=cover` is required for `env(safe-area-inset-*)` to work on notched devices. `globals.css` exposes these as `--safe-top` and `--safe-bottom`; the app root pads its top with `--safe-top`, and `FloatingNav`, `Sheet`, `ToastContainer`, and the screens pad their bottom against `--safe-bottom`.

## Styling

Hybrid Tailwind CSS 4: utility classes (`bg-surface`, `text-ink`, `border-line`, ...) for static styling, with inline `style={{}}` reserved for computed or bespoke values (e.g. `color-mix(in srgb, ${habit.hex} ...)`, dynamic positions, animated offsets).

The design tokens are runtime CSS custom properties defined on `<html>` (and `html[data-mode='...']`). An `@theme inline` block in `src/styles/globals.css` bridges each token into a Tailwind color so utilities resolve to the same `var()` and stay theme- and accent-aware automatically:

```css
@theme inline {
  --color-surface: var(--surface);
  --color-ink: var(--ink);
  --color-accent: var(--accent);
  /* ... */
}
```

Core tokens: `--bg`, `--surface` / `--surface-2` / `--surface-3`, `--line` / `--line-strong`, `--ink` / `--ink-2` / `--ink-3`, `--accent` / `--accent-2` / `--accent-soft`, `--on-accent`, `--good`, `--ring-track`. The accent trio is overwritten at runtime in `App.tsx` from the user's chosen accent color.

### Themes

Three palettes are selected by a `data-mode` attribute on `<html>`:

| Mode         | When                     |
| ------------ | ------------------------ |
| `light`      | default (no `data-mode`) |
| `dark-oled`  | dark mode on mobile      |
| `dark-slate` | dark mode on desktop     |

The mode is computed in `App.tsx` from the theme setting (`auto` follows `prefers-color-scheme`) and the desktop/mobile breakpoint.

The display font is **Geist**, loaded from Google Fonts.

The `.scrollable` utility class enables momentum scrolling for touch devices.
